## Month 2, Week 1 (Week 5): Next.js 基礎架構

### 1. App Router：檔案即路由

#### 🗺️ 比喻：市政廳的地圖
Next.js 的 `app/` 資料夾就像是一座城市的**市政廳**。
- **資料夾 = 行政區 (URL 路徑)**：
    - 你在 `app/` 裡蓋了一個資料夾叫 `about/`，你的網站就自動有了 `yoursite.com/about` 這條路。
    - 蓋了 `blog/`，就有 `yoursite.com/blog`。
    - 完全不用寫程式設定路由，**檔案結構就是地圖**。
- **`page.tsx` = 主要展示廳**：
    - 每個區(資料夾)裡面的 `page.tsx` 就是遊客走進去後看到的**主要畫面**。
- **`layout.tsx` = 建築骨架**：
    - 定義那些「不會變」的部分（如導航列、頁尾）。就像市政廳的共同走廊，不管去哪一間辦公室，走廊都在。

### 2. Server vs Client Components

#### 🍽️ 比喻：精緻餐廳 (The Restaurant)
想像你的網頁應用程式是一間餐廳：

- **Server Components (伺服器組件) = 廚房與主廚**
    - **位置**：後場（伺服器端）。
    - **特長**：擁有所有食材（資料庫）、且運算能力強大。
    - **工作**：把菜做好、擺盤漂亮 (Render HTML)，然後端出去。
    - **限制**：這裡沒有客人，所以不能「按讚」、不能「填表單」（因為還沒送到客人桌上）。
    - **預設**：Next.js 預設所有組件都是廚房做的。

- **Client Components (客戶端組件) = 餐桌與調味料**
    - **位置**：客人的餐桌（瀏覽器端）。
    - **特長**：可以和客人互動（點擊、輸入）。
    - **指令**：當你需要互動功能（比如 `onClick`），你必須在檔案最上面寫一行指令：`"use client"`。這就像是在餐桌上放一個服務鈴。
    - **工作**：處理那些需要即時反應的小事（如摺疊選單、輪播圖）。

## Month 2, Week 2 (Week 6): API 與通訊

### 1. 什麼是 API？

#### 🤵 比喻：餐廳服務生 (The Waiter)
- **你 (前端/客人)**：想吃牛排，但你不能直接衝進廚房自己煎。
- **廚房 (後端/資料庫)**：有食材和爐具，但不知道客人要什麼。
- **API (服務生)**：
    1.  你拿著**菜單 (API 文件)** 點餐。
    2.  服務生記下你的訂單 **(Request)**：「牛排、五分熟」。
    3.  送進廚房。
    4.  廚房做好後，服務生把牛排端給你 **(Response)**。

### 2. 環境變數 (.env)

#### 🔐 比喻：保險箱密碼
有些東西不能寫在程式碼(公開的菜單)裡，比如資料庫的密碼、API 金鑰。
- 我們把它們寫在 `.env` 檔案裡，這個檔案就像是**經理的私人筆記本**，絕對不會給客人看（不會上傳到 GitHub）。

### 3. Git 與 GitHub

#### ⏳ 比喻：時光機 (Time Machine)
- **Git (時光機)**：可以幫你把專案存檔。
    - `git commit`：建立一個「存檔點」。無論你之後怎麼改壞，隨時可以按「讀取進度」回到這一天。
- **GitHub (時光機博物館)**：
    - 把你的存檔點上傳到雲端，讓大家參觀，或者讓未來的自己下載回來。

## Month 2, Week 3 & 4 (Week 7-8): 部署 (Showtime)

### 1. 部署流程

#### 📚 比喻：出版一本書
- **開發 (Developing)** = **寫手稿**。你在家裡寫，只有看得到。
- **建置 (Building)** = **排版印刷**。機器把你的手稿變成精美的印刷書（最佳化的程式碼）。
- **部署 (Deploying)** = **上架書店**。把他送到全世界的架上（Vercel 的伺服器）。

### 2. Vercel 的魔法
如果你用 Next.js，Vercel 就像是**最強出版商**。
- 你只要把手稿 (程式碼) 推送到 GitHub。
- Vercel 會自動偵測到：「喔！有新稿子！」。
- 它自動幫你排版、印刷、上架。
- 幾分鐘後，它給你一個網址 (URL) —— 你的書正式出版了。
